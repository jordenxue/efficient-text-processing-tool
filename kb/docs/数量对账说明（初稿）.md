# 数量对账说明（初稿）

## 一、目的

本说明用于把当前工程里与数量相关的 3 个关键口径解释清楚：

1. `chunk_total = 1722`
2. `canon_input_count = 1722`
3. `style_input_count = 1729`

重点结论是：`style_input_count = 1729` 表示 `state/processed_style.jsonl` 的记录总行数，不表示有 1729 个唯一 chunk。

## 二、检查对象

本说明直接检查以下真实文件：

1. `state/processed_canon.jsonl`
2. `state/processed_style.jsonl`

## 三、Confirmed（可直接从文件证明）

### 1. canon 口径

1. `state/processed_canon.jsonl` 总行数 = `1722`
2. `state/processed_canon.jsonl` 中唯一 `chunk_id` 数 = `1722`
3. `state/processed_canon.jsonl` 中重复 `chunk_id` 数 = `0`
4. `state/processed_canon.jsonl` 的状态分布为：
   - `success = 1716`
   - `structural_chunk = 6`

结论：

- `canon_input_count = 1722` 与 `chunk_total = 1722` 是对得上的。
- 从 `processed_canon.jsonl` 看，canon 这一层没有把同一个 `chunk_id` 重复记成多行。

### 2. style 口径

1. `state/processed_style.jsonl` 总行数 = `1729`
2. `state/processed_style.jsonl` 中唯一 `chunk_id` 数 = `1722`
3. `state/processed_style.jsonl` 中重复 `chunk_id` 数 = `7`
4. 这 7 个重复 `chunk_id` 一共多出 `7` 行额外记录
5. `state/processed_style.jsonl` 的状态分布为：
   - `success = 1707`
   - `failed = 15`
   - `structural_chunk = 7`

结论：

- `style_input_count = 1729` 不能理解为“有 1729 个唯一 chunk”。
- 从唯一 `chunk_id` 口径看，style 处理的仍然是 `1722` 个 chunk。
- `1729 - 1722 = 7` 的差额，来自重复记录，而不是新增唯一 chunk。

### 3. style 重复记录的直接构成

可直接从 `processed_style.jsonl` 看到，7 个重复 `chunk_id` 分成两类：

1. `failed -> success` 的 6 组：
   - `全民魔女1994_0006`
   - `全民魔女1994_0026`
   - `全民魔女1994_0035`
   - `全民魔女1994_0038`
   - `全民魔女1994_0051`
   - `全民魔女1994_0053`
2. `structural_chunk` 被重复记录 2 次的 1 组：
   - `全民魔女1994_0001`

因此，`style_input_count = 1729` 比 `chunk_total = 1722` 多出的 7 行，可以直接拆解为：

1. 6 行来自 6 组 `failed -> success`
2. 1 行来自 `全民魔女1994_0001` 的重复 `structural_chunk` 记录

### 4. style 中仍然只保留 failed 的 chunk_id

`processed_style.jsonl` 中还有 9 个 `chunk_id` 只出现 `failed`，没有对应的 `success`：

1. `全民魔女1994_0551`
2. `全民魔女1994_0588`
3. `全民魔女1994_0665`
4. `全民魔女1994_0932`
5. `全民魔女1994_1332`
6. `全民魔女1994_1375`
7. `全民魔女1994_1399`
8. `全民魔女1994_1588`
9. `全民魔女1994_1714`

这 9 个 only-failed `chunk_id` 与当前工程里“style 剩余 9 个 hard holes 已归档”的基线数量是一致的。

## 四、Inference（合理推断，但不能仅凭这两个文件完全证明）

### 1. 对 1722 vs 1729 的最合理解释

最合理的解释是：

1. `chunk_total = 1722` 表示冻结后的正式 chunk 输入集规模
2. `canon_input_count = 1722` 表示 canon 处理记录与冻结输入集一一对应
3. `style_input_count = 1729` 表示 `processed_style.jsonl` 的原始处理记录总行数

也就是说，`style_input_count` 更接近“style 处理日志条目数”，而不是“style 唯一输入 chunk 数”。

### 2. 重复主要来自什么口径

从重复记录本身看：

1. 6 组 `failed -> success` 更像是 retry 或重复处理后的补成功
2. `全民魔女1994_0001` 的两条 `structural_chunk` 更像是重复写入或重复处理记录

因此，若只针对“1722 vs 1729 的 7 行差额”做解释，最稳妥的表述是：

- **主要来自 retry / 重复处理口径**
- **不能把这 7 行差额直接归因为 backfill**

原因是：

1. 这 7 组重复记录本身没有显式 `backfill=true` 之类字段
2. 这 7 组重复记录都出现在 `2026-04-11` 的时间戳中
3. 仅从这两个 JSONL 文件，无法把这 7 行差额直接证明为 backfill 写入

### 3. backfill 与当前数量差额的关系

可以合理推断，style 整体历史中确实存在 backfill 阶段，因为项目状态文件与 checkpoint 已明确记录“两轮 backfill 已完成”。

但若只问“为什么 `style_input_count = 1729` 比 `1722` 多 7”，目前最直接、最可证明的答案仍然是：

- 多出来的是重复记录行
- 不是多出来 7 个唯一 chunk
- 这 7 行重复主要表现为 retry / 重复处理痕迹

## 五、一句话结论

`chunk_total = 1722`、`canon_input_count = 1722`、`style_input_count = 1729` 这三个数并不矛盾；真正需要区分的是：

- `1722` 是冻结输入集 / 唯一 `chunk_id` 口径
- `1729` 是 `processed_style.jsonl` 的总记录行数口径

所以，`style_input_count = 1729` 绝不能误写成“style 有 1729 个唯一 chunk”。  
更准确的说法是：style 仍然对应 `1722` 个唯一 chunk，只是处理日志中多出了 7 条重复记录。
