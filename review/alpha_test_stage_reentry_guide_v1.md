# 隔离区文件回迁指南 v1

> 本文件用于约束后续若从 `alpha_test_stage/` 回迁文件到主项目时，应如何操作。
> 回迁不是默认动作，必须经过重新审查。

## 1. 什么时候允许回迁

只有在以下条件同时满足时，才允许考虑回迁：

1. 人类审核认为该文件确实属于项目骨架或长期需要保留的结构材料
2. 该文件不再只是测试产物、运行态文件或阶段性实验材料
3. 该文件的资产定位已经清楚，不会混淆四层边界

## 2. 谁来拍板

- 最终由人类维护者拍板。
- AI 可以提出“建议回迁”，但不能自行把隔离区文件恢复到主项目骨架区。

## 3. 回迁前必须检查什么

回迁前至少检查：

1. 文件是否真的属于骨架，而不是“看起来有用”的实验产物
2. 非脚本文件是否具备合法层级 Tag
3. 文件口径是否最新，是否仍含旧表述
4. 是否含本地路径、运行日志、敏感运行态信息
5. 是否会覆盖主项目中现有更稳定的文件

## 4. 回迁时怎么做

- 回迁时应保留原路径结构。
- 若原目标路径已有文件，先做人类确认，不得直接静默覆盖。
- 若文件只是“可参考”，不等于必须回主项目；可继续留在隔离区。

## 5. 回迁后还要检查什么

回迁完成后，应重新检查：

1. 该文件是否应进入 Git 白名单
2. 相关白名单 / 隔离清单是否需要更新
3. `.gitignore` 是否需要同步调整
4. 入口说明或真源地图是否需要补充该文件的新定位

## 6. 核心原则

- 回迁不是默认动作。
- 回迁前必须重新检查 Tag、文件定位与骨架属性。
- 实验产物不能因为“看起来有用”就直接回主干。
