# 条件生成指引容器 v0.1 草案

> 本文件用于定义“条件实例资产”的容器级草案与最小记录要求入口。
> 本文件是待审草案，不直接等于正式 `instantiation_mold` 真源。

## 1. 目标

- 为未来“按条件生成的实例资产”预留统一容器。
- 只定义容器级结构与最小记录要求入口。
- 不在本轮冻结具体条件资产类型的详细 schema。

## 2. 当前建议的容器定位

推荐作为未来新小说工作区内的候选挂点：

```text
kb/conditional_assets/
```

当前说明：

- 这是候选位置，不代表本项目已经重构到该路径。
- 该容器只在识别到相应条件时按需启用，不要求每本小说在首轮就填满内容。

## 3. 容器级最小要求

未来若启用该容器，至少应满足：

1. 有统一入口，能说明“该容器为什么存在”
2. 能记录每个条件资产是因什么条件触发生成
3. 能记录该资产与哪些长期实例资产相关联
4. 能挂接审阅 / 收口入口
5. 能追溯其来源范围，而不是只留最终结果

## 4. 每个条件实例资产的最小记录要求

当前建议至少记录以下内容：

- `trigger_condition`：为什么会生成该资产
- `scope`：它作用于哪类对象或哪段范围
- `related_paths`：与哪些长期实例资产相关
- `source_scope`：依据了哪些材料范围
- `review_entrypoint`：后续在哪里审阅与收口

说明：

- 本轮只冻结“最小记录要求”。
- 不冻结具体字段格式、枚举值或 per-type 细 schema。

## 5. 本轮不冻结的内容

- 条件资产的固定文件名
- 条件资产的固定 JSON schema
- 条件资产的类型枚举
- 条件资产是否必须拆子目录
- 条件资产与主卡 / sidecar 的自动生成逻辑

## 6. 当前结论

本轮已经可以先把“条件生成指引容器”抽象成：

- 一个未来位于小说实例层内部的按需容器位
- 一组跨小说复用的最小记录要求

这足以支撑后续正式 mold 定稿与结构重构，但还不应在本轮直接冻结成正式模具真源。
