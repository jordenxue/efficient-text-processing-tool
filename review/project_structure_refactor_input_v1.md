# 项目结构重构输入分析 v1

> 本文件用于给后续“项目结构重构”提供输入材料。
> 本文件不是正式重构方案，不在本轮执行目录迁移或大规模文件移动。

## 1. 当前已可确认的稳定长期定位

### 1.1 `universal/`

- `universal/specs/` 已具备明确的 base 真源定位。
- `universal/sop/` 已开始承载跨小说 SOP 与验收清单，方向上也更接近 base。
- 这说明 `universal/` 作为 base 层承载区已经基本成立。

### 1.2 `kb/characters/`

- `kb/characters/` 已明确为小说级长期角色资产方向。
- `kb/characters/江涵/` 已提供主卡 + sidecars 的结构样板。
- 这说明“角色长期资产在 `kb/characters/` 下落地”的方向已具备稳定性。

### 1.3 `outputs/`

- `outputs/generation/`、`outputs/ab_round3/` 当前都应视为实验消费层。
- 这些目录适合继续承载 payload、prompt、pilot 输出与日志。
- 它们不应被误写成长期真值或正式 mold 真源。

## 2. 后续重构时应重点考虑的边界问题

### 2.1 `universal/specs/` 与项目级协作文档的关系

- `universal/specs/` 适合继续承载正式规范与协议。
- 根目录下的 `AI_ROLES.md`、`STATE.json`、`CURRENT_STATE.md`、`NEXT_ACTION.md`、`DECISIONS.md` 更像项目运行态协作文档。
- 后续重构时应避免把“跨小说固定规范”与“当前项目运行状态”混放成同一类资产。

### 2.2 `kb/characters/` 与未来 mold / novel_instance 的边界

- `kb/characters/` 当前应继续保留为具体小说长期资产落点。
- 未来正式 mold 不应直接落在 `kb/characters/` 中，而应由 mold 生成其骨架，再落入小说实例层。
- 也就是说，`kb/characters/` 适合承接实例结果，不适合承接 mold 真源。

### 2.3 `review/` 内部对象的差异

- `review/` 目前混合了阶段诊断、清理隔离、重构分析、Tag 说明、实验复审等多类文件。
- 其中一部分是阶段性协作文档，另一部分则是可反复引用的分析输入。
- 后续重构时应考虑把“实验复审”“结构分析”“清理审计”进一步分开，而不是长期混放。

### 2.4 `outputs/` 的定位约束

- `outputs/` 下内容应继续视为消费产物、运行产物或实验输入。
- 即使其中某些材料很有价值，也应先提炼出规范或模具，再决定是否进入长期层。
- 后续重构时要防止 `outputs/` 中的高频使用文件被误当成真值资产。

## 3. 哪些内容适合后续迁入 mold 层

当前最适合抽象为 mold 的，不是具体内容，而是以下结构：

1. 新小说独立工作区总文件夹骨架
2. `kb/characters/` 的角色资产容器骨架
3. 主卡 + sidecars 的存在性骨架
4. `review/` 与 `logs/` 的基础挂接位
5. 条件实例资产的容器位与最小记录要求入口
6. `raw / chunks / state / outputs/canon / outputs/style` 等基础底料容器位

这些对象具备“跨小说复用、结构稳定、与具体小说内容解耦”的特征。

## 4. 哪些内容当前仍应留在 experimental

- round2 / round3 的复审文件与诊断文件
- round3 prefix / prompt / checkpoint / guardrail / pilot 输出
- 世界规则补充卡、称呼规则补充卡、旧角色卡样板
- 当前项目运行态协作文档中的“阶段推进内容”
- 各类清理清单、迁移报告、归位盘点、Tag 落地说明与本文件自身

原因：

- 这些对象仍然带有阶段性、实验性或项目运行态属性。
- 它们可以为后续定稿提供证据，但暂不应被误判为稳定真源。

## 5. 哪些内容仍待验证，不宜过早固化

- 角色主卡内部更细的 generation-facing 字段簇
- sidecar 的更细拆分方式
- 条件实例资产的具体类型体系
- `review/` 的最终分区方式
- 某些历史 schema 与历史测试集是否继续保留为 base

## 6. 对后续结构重构最有用的直接输入

后续重构时，可直接以本轮形成的三类输入为起点：

1. “稳定核心实例模具 v0.1 抽取草案”
2. “条件生成指引容器 v0.1 草案”
3. 当前资产四层归位表与 Tag 落地说明

它们共同回答了三件事：

- 哪些结构已经足够稳定，可以抽象
- 哪些内容只是阶段材料，不应提前固化
- 哪些目录边界在重构时最需要优先澄清

## 7. 当前结论

- 当前已经可以进入“先重构结构表达，再决定是否脚本化”的阶段。
- 但本轮还不应直接移动大量文件，也不应提前宣布 mold 已终局定稿。
- 后续结构重构应优先围绕“base 区更清楚、mold 候选可落位、novel_instance 更纯、experimental 不冒充真值”展开。
