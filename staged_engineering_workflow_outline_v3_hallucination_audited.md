# 阶段性工程版流程文档大纲 v3（幻觉审计修订版）

## 1. 文档定位

### 1.1 目的
本文件用于定义当前阶段的小说改写/续写工程化工作流，作为从"高精度低召回蒸馏底座"向"可持续产数、可回审、可迭代的数据工厂"过渡的执行纲要。

### 1.2 当前适用范围
当前工作流同时覆盖以下三类任务：

- 续写
- 改写
- 续写+改写混合

**说明：**
当前阶段不设独立的任务分流模块（Step 0）。三类任务通过规划层内部的 `task_mode` 字段区分（详见 Step 5），共用同一套 plan 模板，不维护三套独立入口。

### 1.3 与现有主链的关系
当前仓库中已有的 `canon/style` 双管线继续保留，作为稳定底座使用，不在本阶段推翻或重写。

### 1.4 本文档不负责的事项
详见 Section 13"当前阶段不做事项"。

---

## 2. 当前阶段总目标

### 2.1 核心目标
将现有"chunk 抽取 + 静态知识底料"流程，升级为一个可以支撑以下能力的阶段性工程系统：

1. 批量产出更高召回的数据
2. 构建可查询、可复审的知识资产
3. 为续写/改写生成提供结构化规划输入
4. 为后续训练与评估提供样本来源
5. 逐步接入一致性审计与图谱化表示

### 2.2 当前阶段的成功标准
本阶段成功不以"理论最先进"为目标，而以"能稳定产出并支撑后续迭代"为目标。

至少应达到：

- 能稳定从原文中提取出比当前更多的实体、关系、事件候选
- 能把这些候选整理成可审计资产
- 能从资产构造出可用的续写、改写任务输入
- 能产出一批可评估的正文候选
- 能对候选执行基础一致性与质量审核

---

## 3. 工程原则

### 3.1 稳定底座优先
现有 `canon/style` 主链为当前稳定底座，不做架构级大改。

### 3.2 新能力先走实验层
所有新增流程、脚本、中间资产、候选结果，先按实验/消费层理解，不直接升格为长期真值。

### 3.3 先产数，后精修
当前优先解决"蒸不出很多数据"和"无法形成闭环"的问题；过度精修单一 prompt 或单一卡片，不是当前优先级。

### 3.4 先单模型多轮，后谈复杂编排
当前阶段默认采用单模型、多 pass、可回审流程。  
multi-agent 无限期延后（详见 Section 13）。

### 3.5 审计多道关卡
一致性与矛盾审计不是一次性步骤，而是在多个阶段插入的守门动作。当前流程中设有以下审计关卡：

- **Step 2 出口**：召回候选轻量一致性门（详见 6.7）
- **Step 4**：知识资产一致性审计（主审计步骤）
- **Step 6**：质量审核中包含约束偏离审计（详见 10.3、10.8）

### 3.6 约束是跑道，不是牢笼
当前工作流强调"结构化约束"，但约束的目标是防穿帮、防漂移、保可控，不是把文本写死。  
当前阶段默认区分两种生成模式：

- `draft_mode`：宽松模式，优先多样性、文学性、备选方向
- `finalize_mode`：严格模式，优先收口、修正、定稿一致性

**注意**：此处"两种模式"仅指人工在不同阶段选择的工作姿态，不涉及强化学习中 exploration / exploitation 的技术含义，不暗示任何概率采样或奖励信号机制。

### 3.7 先宽后严
默认执行节奏为：

1. 第一轮规划与生成优先采用 `draft_mode`，放出 3–5 个差异明显的候选
2. 人工或审核模块选定方向后，再切换为 `finalize_mode` 做精修与定稿

---

## 4. 总体流程总览

当前阶段工作流采用以下六步结构：

1. 原文分段与基础蒸馏
2. 高召回补充与候选扩表（含出口审计门）
3. 知识资产构建
4. 一致性审计（第一次）
5. 规划与风格化正文生成
6. 质量审核与迭代（含约束偏离审计）

---

## 5. Step 1：原文分段与基础蒸馏

### 5.1 目标
将原始小说文本切分为可处理的基本单元，并生成最基础、最稳定的结构化底料。

### 5.2 当前职责
本步骤负责：

- 原文导入
- chunk 切分
- chunk meta 生成
- `canon/style` 抽取
- 基础日志与状态记录

### 5.3 输入
- 原始正文文本
- 章节标题、卷标记（若有）
- 现有分块规则
- 附属 meta 文件（`*.meta.json`）

### 5.4 输出
- `chunks/*.txt`
- `chunks/*.meta.json`
- `outputs/canon/*.json`
- `outputs/style/*.json`
- 对应 state、log、summary 文件

### 5.5 当前问题
- chunk-local，跨 chunk 信息缺失严重
- 实体、关系、长列表召回不足
- style 抽取仍偏标签化，难以直接服务续写、改写深层约束

### 5.6 当前阶段要求
- 不推翻该步骤
- 只允许做低风险增强
- 保持对后续步骤的兼容性

---

## 6. Step 2：高召回补充与候选扩表

### 6.1 目标
在不引入 multi-agent 的前提下，补足现有基础抽取的召回短板。

### 6.2 核心思路
基于现有 `canon/style/kb/chunks` 做轻量 recall loop：

1. 生成初始候选
2. 回查 chunks
3. 用别名、共现、关系词、地点词等扩一轮
4. 记录新增候选与证据
5. 迭代至达到轮数上限或增量候选数降至阈值以下

### 6.3 当前优先覆盖对象
- 人物扩表
- 地点扩表
- 基础关系候选
- 事件候选补充

### 6.4 输入
- `canon/style` 输出
- 角色、地点已有索引
- chunks 与 chunk meta
- alias 数据、appearance 数据（若有）

### 6.5 输出
- recall candidates
- evidence chunk lists
- alias candidates
- relation candidate lists
- low-confidence、unresolved candidate logs

### 6.6 当前阶段要求
详见 Section 13"当前阶段不做事项"中的相关条目。本步骤先追求"可解释的增量召回"。

### 6.7 出口审计门
本步骤输出在进入 Step 3 之前，须通过轻量一致性检查：

- 新增候选之间是否存在明显互斥（如同一候选同时关联矛盾身份）
- 新增别名候选是否与已确认别名产生冲突
- 是否有无证据支撑的孤立候选

此门不要求穷尽审计，仅做快速拦截明显错误，防止脏候选污染 Step 3 聚合。

### 6.8 验收重点
- 是否真的找到了更多候选
- 是否带证据
- 是否可人工复核
- 是否明显优于只看 `canon/style`

---

## 7. Step 3：知识资产构建

### 7.1 目标
把分散在各个 chunk 和 recall 候选中的信息，聚合为可持续维护的知识资产。

### 7.2 本步骤负责的对象类型
- 角色表
- 地点表
- 事件表
- 关系表
- 时间线候选
- 证据映射
- scene/function 候选（**待定义**：指场景的叙事功能标注，具体 schema 待后续设计，参见 Section 15）
- character state 候选（**待定义**：指角色在特定时间点的状态快照，具体 schema 待后续设计，参见 Section 15）

### 7.3 图谱接口
本步骤同时负责产出 graph-ready triples（三元组候选），作为后续图谱层的输入材料。triples 在本步骤产出，但图谱的组装、可视化与导出由图谱增强层负责（详见 Section 11）。

**覆盖范围说明**：当前三元组仅覆盖事实型关系（如"A 是 B 的父亲""A 在 C 地点出现"）。时序状态（如"角色 A 在第 N 卷前认为 B 已死"）、视角限制、情感弧线等叙事型知识不适合直接用三元组表示，其表示方式待后续设计。

### 7.4 输入
- Step 1 的基础抽取结果
- Step 2 的召回补充结果（已通过出口审计门）
- 既有 kb 资产
- alias、appearance、relation 等辅助文件

### 7.5 输出
- entity-centric aggregates
- relation aggregates
- event chains
- timeline candidates
- graph-ready triples（仅事实型关系）

### 7.6 当前阶段要求
- 先以"可查、可审、可继续加工"为目标
- 不急于一次性定成最终 schema
- 长期真值与实验候选要分开存放

### 7.7 风险
- 候选和真值混层
- 低置信关系被错误固化
- 同名、别名未归并干净

---

## 8. Step 4：一致性审计（第一次）

### 8.1 目标
在进入规划与生成之前，对知识资产做第一轮一致性检查。

### 8.2 本步骤重点检查
- 人物身份冲突
- 别名冲突
- 时间线明显冲突
- 地点归属冲突
- 关系自相矛盾
- 低置信但高影响的资产节点

### 8.3 高危区定向排雷
本步骤引入 `risk_level` 概念，对高危输入片段倾斜更多审计预算。

默认高危条件包括：

- 叙事中段 chunk / 中段场景
- 时间推进密集段
- 多角色并行互动段
- 高转折、高信息密度段
- 大量引用既有关系或既有事实的续写段

建议风险等级：

- `L1_low`
- `L2_medium`
- `L3_high`

其中 `L3_high` 默认触发更严格的审计 prompt、更高 token 配额或更细粒度 evidence 检查。

### 8.4 输入
- Step 3 构建出的知识资产
- 原文证据映射
- recall 候选日志
- 当前片段风险等级

### 8.5 输出
- conflict reports
- unresolved issues list
- review-needed items
- safe-to-use asset subset
- high-risk audit notes

### 8.6 当前阶段要求
- 第一版先做基础冲突检测
- 高危区优先审计，而不是对所有片段平均施力
- 审计输出必须带证据指向或冲突原因说明

### 8.7 作用
本步骤不是终局清洗，而是进入生成前的第一道风险收口。

---

## 9. Step 5：规划与风格化正文生成

本步骤合并"结构化规划"与"风格化正文渲染"两个阶段为一步，内部分为 Phase A（规划）和 Phase B（生成）两个连续 phase。合并原因：插槽式骨架（Phase A 输出）和逐槽填充（Phase B 输入）在执行层面为同一 prompt chain 的前后段，中间无需强制人工断点。若后续发现 Phase A 输出确需人工 checkpoint，可再拆回两步。

### Phase A：结构化规划

#### 9.1 目标
把"知识资产 + 当前任务上下文"转成结构化的写作计划。

#### 9.2 任务模式处理
三类任务（续写、改写、混合）通过统一的 plan 模板中的 `task_mode` 字段区分，不维护三套独立 plan 产物。

`task_mode` 取值：
- `continuation`
- `rewrite`
- `hybrid`

不同 `task_mode` 影响的是 plan 模板内部的字段启用规则（如 `rewrite_intent` 仅在 `rewrite` / `hybrid` 下启用），而非模板结构本身。

#### 9.3 本 phase 负责生成的中间物
- 统一格式的 plan（含 `task_mode` 字段）
- scene cards
- event sequence
- continuity constraints（**待定义**：指规划层输出的、生成层必须遵守的一致性约束条目；具体数据结构与 schema 待后续设计，参见 Section 15。此概念为本流程中 Phase A 与 Phase B、以及 Step 5 与 Step 6 之间的核心传递物，应优先落地定义。）
- must preserve、must avoid 列表
- 带锚点的 slot skeleton

#### 9.4 插槽式骨架
规划层不只输出"情节点列表"，还应输出可供 Phase B 填充的插槽式骨架，例如：

- `[ACTION_SLOT]`
- `[DIALOGUE_SLOT]`
- `[DESCRIPTION_SLOT]`
- `[TRANSITION_SLOT]`
- `[INNER_STATE_SLOT]`

每个 slot 需附最小说明，例如：
- 本槽位的目标
- 必须命中的信息点
- 允许的自由度
- 关联的 continuity constraints

#### 9.5 约束设计原则
`must preserve / must avoid` 不应写成机械死规矩，而应尽量写成"创作杠杆"。

优先写法：
- 保留感知顺序
- 保留信息暴露顺序
- 保留角色反应方式
- 保留关键视角与情绪基调

避免写法：
- 机械句长限制
- 机械字数限制
- 机械"不得出现 X 修辞"之类的僵硬禁令

#### 9.6 Phase A 输入
- 通过第一次审计的知识资产
- 用户任务上下文
- 原文上下文片段
- 风格资产（若已生成）
- 当前剧情锚点、rewrite intent（若有）
- 当前模式（`draft_mode` / `finalize_mode`）

#### 9.7 Phase A 输出
- 结构化 plan（含 `task_mode`）
- 场景序列
- 带 slot 的 prose skeleton
- 风险提示
- 待确认点列表

### Phase B：风格化正文生成

#### 9.8 目标
将 Phase A 产出的结构化内容，渲染为可阅读的正文候选。

#### 9.9 建议的单模型多轮生成方式
第一版建议尝试"填槽 + 综合"的去 agent 流程：

1. 先填 `ACTION_SLOT`
2. 再填 `DIALOGUE_SLOT`
3. 再填 `DESCRIPTION_SLOT / INNER_STATE_SLOT`
4. 最后做 synthesis / polish

**注意**：上述填充顺序与分槽策略为初始假设，尚无已知实验验证其对小说写作场景的有效性。Skeleton-of-Thought（Ning et al., ICLR 2024）等相关工作主要关注推理加速而非创意写作质量，且在 writing 类任务上效果有限。本方案是否优于直接基于大纲的自由生成，需要在第一轮实验中对比验证。

#### 9.10 Phase B 输入
- Phase A 的结构化 plan 与 prose skeleton
- 原文上下文
- continuity constraints
- 风格资产、style guidance
- 角色、关系、地点等必要约束
- 当前模式（`draft_mode` / `finalize_mode`）

#### 9.11 Phase B 输出
- candidate drafts
- multiple candidate variants
- generation logs
- candidate metadata（含约束偏离记录，详见 9.13）

#### 9.12 当前阶段要求
- 规划侧：先追求"可消费的结构化计划"，不要求一次性极致智能；规划结果必须显式暴露不确定点；在 `draft_mode` 下允许候选分支，`finalize_mode` 下更靠近已确认线稿
- 生成侧：优先支持多候选，而不是单答案；候选之间应有差异策略（如保守、贴风格、推进剧情）；第一轮优先 `draft_mode`，后续定稿再切 `finalize_mode`；不要求一次产出最终成品

#### 9.13 生成时的默认文学性原则
Phase B 的默认指引应强调：

- 在遵循约束前提下，优先保证文本的自然流畅度与文学性
- 若约束与文采发生轻微冲突，允许轻微偏离约束
- **若发生偏离，必须在候选元数据中显式记录偏离点与偏离原因**

此记录将在 Step 6 约束偏离审计中被核查（详见 10.8）。

#### 9.14 当前风险
- 生成绕开测试点
- 风格表面像但叙事推进不对
- 连续性约束未被真正消费
- 候选太像，无法形成有效偏好筛选
- 约束写得过死，导致文本发僵

---

## 10. Step 6：质量审核与迭代

### 10.1 目标
对正文候选进行质量评估、偏好筛选和必要的回修。

### 10.2 本步骤负责
- 基础一致性检查
- 风格贴合度检查
- 人物不崩检查
- 候选排序
- winner、loser 标注
- error tags 记录
- 回修建议生成

### 10.3 主线机制：pairwise comparative review
当前主线不采用完整 debate pipeline，而采用更轻量、可控的成对比较审核。

比较时应显式输出：

- 候选 A 的优势
- 候选 B 的优势
- 两者各自的 error tags
- 最终 winner / loser
- 偏好理由
- 是否需要合并两者优点再生成一轮

### 10.4 实验接口：对抗式多轮比较评估
保留一个实验接口，用于后续探索比 pairwise comparative review 更深入的候选评估方式。初步构想为：让两个 LLM 角色分别为两个候选辩护，经过多轮交锋后由人工或第三方 judge 裁决。

**注意**：此方案为概念占位符，当前无已知先例将此类协议用于创意写作候选评估。是否有效、是否显著优于简单 pairwise review，均需从零验证。不进入当前主线。

### 10.5 输入
- 正文候选（含约束偏离元数据）
- 规划结果
- 审计结果
- 原文上下文
- 风格与连续性约束
- 当前模式（`draft_mode` / `finalize_mode`）

### 10.6 输出
- 审核报告
- preference pairs
- error annotations
- revised candidate suggestions
- 可入训练集的样本草案
- 约束偏离审计报告

### 10.7 当前阶段要求
- 第一版允许半自动
- 不要求 judge 完全替代人工
- 审核输出要能反哺后续 prompt、schema、workflow 迭代
- `draft_mode` 优先保留多样性；`finalize_mode` 优先收口和消除穿帮

### 10.8 约束偏离审计
本步骤须包含一道显式的约束偏离审计，核查 Step 5 Phase B 输出中的偏离记录：

- 每条偏离是否确实因文学性需要而产生
- 偏离是否触及 continuity constraints 或 must preserve 列表中的硬约束
- 偏离是否引入了新的一致性问题

此审计与 pairwise review 并行执行，不是其子步骤。目的是防止"允许轻微偏离"在多轮迭代中无声积累为系统性穿帮。

### 10.9 与后续训练的关系
本步骤的结果，将作为后续：

- SFT 样本
- 偏好对样本
- 评估样本

的重要来源之一。

## 11. 图谱层的阶段性位置

### 11.1 当前定位
图谱层不是当前主流程中的独立步骤，而是 **Step 3 之后的增强层**。Step 3 负责产出 graph-ready triples（原始三元组候选），图谱层负责将这些 triples 组装为图结构并提供可编辑导出。

### 11.2 当前阶段目标
先做图结构组装与可编辑导出，不急于图数据库正式化。

### 11.3 第一版形式
- 用 `networkx` 做原型
- 支持基础节点、边构建
- 除图对象本体外，强制导出可编辑表示：
  - CSV triples
  - Markdown triples table
- 第一版优先验证"可编辑性与可干预性"，其次才是图查询收益

### 11.4 当前优先价值
当前阶段中，图谱层的首要价值不是"更炫的查询"，而是：

- 给人工 review 提供直观抓手
- 允许直接修改 triples
- 把人工修改结果回流给下一轮规划与生成
- 为后续更复杂的连续性控制留接口

### 11.5 当前不做
详见 Section 13。

## 12. 当前阶段的资产分层与状态标记

### 12.1 说明
项目正式资产层级严格只有四层。当前阶段在四层之上，仅补充一个 `stability` 状态轴，用于区分同层内资产的成熟度。

### 12.2 正式资产层级（唯一层级轴）

#### `base`
用于承载跨小说稳定复用、且应由人类控制版本的规则、协议、模板、通用能力。

当前阶段中，以下对象若进入长期稳定状态，应归 `base`：

- 跨小说通用 workflow 规范
- 通用协议文档
- 通用 schema
- 通用 prompt 模板
- 通用评估协议
- 通用工厂脚本能力
- 通用图谱构建协议
- 通用 recall loop 协议或接口规范

#### `instantiation_mold`
用于承载跨小说复用的空骨架或实例化模板，而不是具体小说内容本身。

当前阶段中，以下对象若被收口为模板，应归 `instantiation_mold`：

- author style asset 模板
- scene card 模板
- character state 模板
- plan 模板（含 `task_mode` 字段）
- preference pair 模板
- graph-ready 输入骨架模板
- 新小说初始化时应复制出的空目录骨架与空文件骨架

#### `novel_instance`
用于承载某一本小说的长期结构化知识资产与基础底料。

当前阶段中，以下对象在通过审阅、确认要长期保留后，应归 `novel_instance`：

- chunk 与 chunk meta
- `canon/style` 结果
- 角色、地点、事件、关系聚合结果
- 时间线候选的已确认部分
- 审核通过的 author style 资产
- 审核通过的 character state、scene function 资产
- graph-ready triples 的已确认部分
- 某本小说专属、需要长期维护的知识资产

#### `experimental`
用于承载当前阶段的实验消费材料、运行中间产物、候选结果、评估结果和阶段性协作文档。

当前阶段中，以下对象默认应先归 `experimental`：

- recall loop 输出
- alias candidates
- unresolved relation candidates
- conflict reports
- plans（含各 `task_mode`）
- candidate drafts
- judge results
- preference pairs
- dataset builds
- graph prototype 输出
- review 文档
- 状态文件
- 一切尚未审核通过的中间资产和消费层产物

### 12.3 稳定性状态（`stability`）
用于标记同层内资产的成熟度：

- `draft`：初稿，未经任何审阅
- `reviewed`：已经过审阅，可使用但非最终态
- `frozen`：已定稿，非经显式流程不得修改

大部分情况下，`asset_layer` + `stability` 两轴已足以定位资产状态。例如：
- `experimental` + `draft` = 未审阅的实验产物
- `novel_instance` + `frozen` = 已定稿的小说知识资产
- `instantiation_mold` + `reviewed` = 已审阅但仍可迭代的跨小说模板

### 12.4 当前阶段的升格规则
为避免候选与真值混层，当前阶段采用以下默认升格规则：

1. 新流程产生的中间资产，默认先归 `experimental`
2. 经审阅确认可长期保留的小说专属资产，升级为 `novel_instance`
3. 经抽象收口、确认跨小说复用的模板骨架，升级为 `instantiation_mold`
4. 经正式定稿、作为跨小说规则、协议、通用能力保留的对象，升级为 `base`

## 13. 当前阶段不做事项

### 13.1 明确后置
- Step 0 任务分流模块（当前通过 `task_mode` 字段在规划层内部处理）
- multi-agent
- 全量图数据库（Neo4j 正式迁移、图谱驱动生成主链、图谱作为唯一真理源）
- RL、reward model 主流程
- 大规模外部工具强依赖接入
- 全历史重跑
- 大而全 RAG 平台
- 复杂 agent 编排

### 13.2 原因
当前优先级是先形成可执行、可产出、可迭代的基础工程闭环，而不是追求系统复杂度。

## 16. 当前版本总结

当前版本工作流的定位不是终局架构，而是一个阶段性工程版本：

- 保住稳定底座
- 补足高召回
- 引入高危区定向审计
- 用插槽式骨架替代直接自由续写（待实验验证有效性）
- 用 draft / finalize 双模式避免过度僵化
- 建立可审计资产
- 在多个关卡插入审计（Step 2 出口、Step 4、Step 6）
- 支撑规划与生成的紧耦合执行
- 形成审核与迭代闭环
- 为后续图谱、训练与实验增强留接口
