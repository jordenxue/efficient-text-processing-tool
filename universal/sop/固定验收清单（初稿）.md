# 固定验收清单（初稿）

> 本文件当前作为 `base` 层验收参考保留；文件名中的“初稿”仅反映历史命名，不单独表示实验层或其他资产层。

一、用途

本清单用于判断一本新书是否已经按项目当前稳定基线，完成了从 chunk 到 query 的第一阶段处理，并具备进入收口归档、复用 SOP 整理或后续小增强的条件。

本清单不用于：

1. 架构级改造验收
2. 新框架接入验收
3. 生成质量主观打分
4. 图数据库或 agent 系统接入验收

二、总原则

1. 先验收主线是否稳定跑通，再讨论小增强。
2. 先验收结构正确，再看质量优化。
3. 任何阶段如出现阻塞，不应靠聊天口头说明替代状态写回。
4. 如某一步未通过，不应直接跳到后续步骤宣称整体通过。
5. 验收以可复核产物为准，不以“感觉差不多”作为通过依据。

三、阶段边界补充（新增）
   
   本清单的验收对象是“第一阶段稳定基线”，也就是：
   1. chunk
   2. canon
   3. style
   4. merge
   5. query
   6. 按需的 section_type 最小审计
   
   本清单默认不验收：
   
   1. continue_writing / rewrite 的 lore fidelity
   2. 生成时的世界规则遵守能力
   3. alias 使用者语境是否被模型正确理解
   4. 双向称呼与关系语域是否被模型正确保持
   5. 候选信息是否会被模型升格为硬事实
   6. 小模型脑补扩写与场景默认补全问题
   
   因此：
   
   1. 通过本清单 = 第一阶段通过
   2. 不等于生成就绪
   3. 若项目目标包含生成，必须进入额外的“生成前预检 / pilot 验证”

三、总通过条件

要判定“第一阶段通过”，至少需要同时满足：

1. sample 阶段通过
2. full run 完成
3. 如存在 style 失败桶，则 backfill 已完成或已明确归档
4. merge 成功产出主知识库
5. query smoke test 通过
6. 状态文件与 checkpoint 已同步到真实状态

四、A. sample 验收

目标：  
确认 chunk → canon → style → merge 的最小链路可跑通，不在全量阶段才暴露致命问题。

通过条件：

1. 样本 chunk 已成功生成并可读
2. canon sample 有有效输出
3. style sample 有有效输出
4. schema 未出现系统性不匹配
5. 没有出现会阻塞 full run 的硬错误
6. 至少形成一个可复核的 sample 级 checkpoint

最低检查项：

1. sample 输入集明确
2. sample 输出目录存在
3. 至少抽查 1 到 3 个 sample 输出，确认不是空文件、不是明显坏 JSON、不是字段严重缺失
4. 当前样本阶段是否已对应 checkpoint
5. `STATE.json` / `CHECKPOINTS.md` 是否反映 sample 已通过

不通过示例：

1. style 连续输出坏 JSON，且没有稳定修补方案
2. canon / style 的 schema 大面积不匹配
3. merge 在 sample 阶段就无法形成有效主产物
4. 状态文件仍显示项目处于未通过样本阶段

五、B. full run 验收

目标：  
确认正式输入集上的主线已经跑完，而不是停留在 sample 或半程状态。

通过条件：

1. chunk 正式输入集已冻结
2. canon full run 完成
3. style full run 完成
4. 全量运行中没有未处理的致命阻塞
5. full run 前后 checkpoint 语义明确

最低检查项：

1. 正式 chunk 总数已记录
2. canon full run 产物存在
3. style full run 产物存在
4. 失败桶、重试、异常是否有记录
5. `STATE.json` 是否明确显示 full run 已完成
6. `CHECKPOINTS.md` 是否已有 full run 对应冻结点

不通过示例：

1. 只有 sample，没有 full run
2. full run 中途停止但被误写成完成
3. chunk 输入集在 full run 期间反复变化
4. 缺少 full run 语义 checkpoint

六、C. backfill 验收

目标：  
确认 style 等失败桶已经收敛到可接受范围，剩余问题被归档而不是悬空。

通过条件：

1. backfill 是否需要，已有明确结论
2. 如需要 backfill，则已完成
3. 剩余失败项已被归档，不再阻塞主线
4. backfill 不改变主线已确认的数据边界

最低检查项：

1. 是否存在 bad chunks / hard holes
2. backfill 轮次是否有记录
3. 剩余问题数量是否明确
4. 剩余问题是否被定性为“不阻塞主线”
5. `STATE.json` / `CHECKPOINTS.md` 是否同步

不通过示例：

1. style 失败桶仍大量未处理
2. backfill 做了一半，没有结果记录
3. 剩余 holes 既未修复也未归档
4. 为了过 backfill 临时放宽 schema 造成 taxonomy 污染

七、D. merge 验收

目标：  
确认全量抽取结果已成功汇总成主知识库，并形成后续 query 可用的稳定基线。

通过条件：

1. merge 成功执行
2. 主知识库产物已生成
3. v2 人物层产物已生成
4. legacy 与真源边界明确
5. merge 结果不需要再回头重构主流

最低检查项：

1. `kb/master_kb.json` 存在
2. `kb/character_appearances_v2.jsonl` 存在
3. `kb/character_profiles_v2.json` 存在
4. `kb/character_query_aliases_v2.json` 存在
5. 旧 `character_index.json` 是否已明确为 legacy
6. `STATE.json` / `DECISIONS.md` 是否反映 v2 为真源

不通过示例：

1. merge 跑完但主产物缺失
2. 旧 `character_index.json` 仍被当成真源
3. 人物层 alias 边界未定
4. merge 结果无法支持后续 query

八、E. query smoke test 验收

目标：  
确认 query 层已经最小可用，不只是“理论上能查”。

通过条件：

1. query.py 可运行
2. 人物查询主路径以 v2 为准
3. canonical / observed alias / query alias 三层解析顺序符合当前基线
4. 至少完成一轮真实 smoke test
5. `appearance_samples` 可用
6. `--json` 结构化输出可用

最低检查项：

1. 用 canonical 名称查询至少通过 1 次
2. 用 observed alias 查询至少通过 1 次
3. 用 query alias 查询至少通过 1 次
4. 至少确认一组应保持分离的名字仍未误合并
5. `--show-appearances` 或等价证据抽样可工作
6. `--json` 返回结构符合预期
7. `STATE.json` / `CHECKPOINTS.md` 是否记录 query 可用

不通过示例：

1. query 仍依赖旧 `character_index.json`
2. query-only 昵称被误写入 observed aliases
3. `appearance_samples` 不可用或污染主事实层
4. `--json` 输出不稳定，无法被脚本消费

九、F. section_type 审计补充验收（按需）

目标：  
当项目发现异质文本混层问题时，确认最小 `section_type` 审计已经正确落地。

这一项不是每本书都必须做，但如果做了，就要通过以下条件：

1. 只落地 `section_type`
2. 未把 `source_scope` 作为独立字段写入 meta.json
3. 扫描清单存在
4. 人工确认清单存在
5. 写入前有 meta 备份
6. 写入后有 verify
7. checkpoint 已建立

最低检查项：

1. `review/scan_candidates.jsonl`
2. `review/confirmed_section_types.jsonl`
3. `review/apply_log.json`
4. `review/meta_backups/`
5. `scripts/apply_section_types.py`
6. `scripts/verify_section_types.py`
7. `checkpoint_section_type_audit_complete_*.zip`

十、G. 状态写回验收

目标：  
确认项目不是“实际上做完了，但文档还停在旧状态”。

通过条件：

1. `STATE.json` 已同步当前真实任务
2. `NEXT_ACTION.md` 与当前主任务一致
3. `CURRENT_STATE.md` 与当前真实基线一致
4. `DECISIONS.md` 已包含新增正式决策
5. `CHECKPOINTS.md` 已包含新增冻结点
6. 若当前任务已完成，状态文件应明确写成 `done`；若尚未选定下一项小增强，可以没有新的活跃工程实现任务，但不能保留伪活跃状态

不通过示例：

1. `STATE.json` 仍停留在旧任务
2. `NEXT_ACTION.md` 和 `CURRENT_STATE.md` 相互矛盾
3. 新 checkpoint 已打包，但未写入 `CHECKPOINTS.md`
4. 决策已经口头形成，但没进 `DECISIONS.md`
5. 收口已完成，但为了维持“有任务在做”而人为保留旧任务的进行中状态

十一、最终结论模板

可用以下模板做最终验收结论：

1. sample：通过 / 未通过
2. full run：通过 / 未通过
3. backfill：通过 / 未通过 / 不适用
4. merge：通过 / 未通过
5. query smoke test：通过 / 未通过
6. section_type 审计：通过 / 未通过 / 不适用
7. 状态写回：通过 / 未通过

最终结论：

* 若以上核心项均通过，则判定“第一阶段通过，可进入收口归档 / SOP 整理 / 后续小增强”
* 若任一核心项未通过，则不得宣称整体通过，必须回到对应阶段处理

十二、一句话收口

固定验收清单的作用，不是增加流程负担，而是防止项目在“似乎做完了”时跳过必要核查；只有通过这套最小验收，才算真正形成可复用、可续接、可审计的稳定基线。



附：本清单应与 `STATE.json`、`NEXT_ACTION.md`、`CHECKPOINTS.md` 联合使用，不单独承担状态真值职责。
