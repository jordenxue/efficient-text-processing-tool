# 项目资产分层与新小说启动总则

> 本文件是项目 `base` 层正式规范文件，用于定义资产分层、新小说启动顺序与资产层级 Tag 规则。

## 1. 资产分层

本项目资产分为四层。

### 1.1 基座（Base）

基座是跨小说固定的项目通用资产，用于定义规则、接口、流程、协议、脚本能力与基础配置。
基座由人类控制版本，AI 默认仅可读取与运行，不得擅自修改。

### 1.2 实例模具（Instantiation Molds）

实例模具是跨小说固定的资产骨架，用于为每一本小说生成结构一致、可维护的实例容器。
实例模具包含两部分：

- **核心模具**：每本小说启动时都应生成的基础骨架。
- **条件生成指引**：用于指导系统在识别到特定结构时，按需生成额外资产，并满足最小记录要求。

### 1.3 小说级实例资产

小说级实例资产是随具体小说变化的长期知识资产。
其首轮内容可由本地 LLM 在基座与实例模具约束下生成，但未经审阅不得视为真值。
该层包括核心实例资产与条件实例资产。
同时，该层内部还应区分：

- **稳定内容**：在较长范围内保持不变的内容。
- **分区内容**：会随阶段、路线、视角、状态或知识范围变化的内容。

### 1.4 实验 / 消费层临时资产

实验 / 消费层资产服务于验证、生成与消费，不构成长期真值资产本体。
该层产物不得自动回流为长期资产。

## 2. 权限与职责

### 2.1 基座权限

基座属于项目稳定层。
AI 默认仅可读取与运行，不得擅自改写；任何修改均应由人类决定。

### 2.2 实例模具权限

实例模具的结构由人类确定。
本地 LLM 可依据实例模具生成小说级实例资产，但不得擅自改写模具源本身。

### 2.3 小说级实例资产职责

小说级实例资产由本地 LLM 首轮生成，再由 Codex、Claude Code 或人工进行审阅、修改、测试与收口。
审核通过后，方可正式入库为长期资产。

### 2.4 实验层职责

实验 / 消费层仅用于验证资产是否可被有效消费，不得反向替代或覆盖长期资产。

## 3. 新小说启动顺序

新小说启动时，应遵循以下顺序：

1. **检查基座**：确认脚本、协议、配置、模板与规范可用。
2. **创建独立工作区**：每本小说必须在独立总文件夹内运行，避免与其他小说混用。
3. **生成实例骨架**：依据核心模具为该小说创建基础资产容器。
4. **准备基础数据**：完成原文导入、切块及通用结构化底料生成。
5. **首轮实例化**：由本地 LLM 依据基座与实例模具生成小说级实例资产，并按条件生成指引决定是否生成条件实例资产。
6. **审阅与收口**：由 Codex、Claude Code 或人工进行多轮审阅、修正、测试与收口。
7. **入库**：将通过审核的内容写入长期资产体系。
8. **进入实验 / 消费层**：仅在资产较稳定后，再进行生成消费验证。

## 4. 条件实例资产原则

条件实例资产不是每本小说都必须存在。
是否生成，应由本地 LLM 根据条件生成指引与小说实际特征判断。
不同小说中的条件实例资产可以采用不同形态，但至少应满足统一的最小记录要求，以保证可追溯、可审核、可挂接。

## 5. 分区变化原则

并非所有资产都应整体分区。
分区应仅作用于真正会随阶段、路线、视角、状态或知识边界变化的字段簇，而不应无差别切碎全部资产。

角色资产中，以下内容应优先考虑分区记录：

- 角色状态
- 角色知识
- 某些关系状态
- 某些能力状态

以下内容默认优先保持稳定记录：

- 身份核心中的稳定部分
- 长期硬边界
- 不依赖阶段变化的基础设定

分区记录的目标，是在不破坏整体可维护性的前提下，准确保存变化内容；
不应以分区替代整体结构，也不应将整份资产无必要地拆散。

## 6. 当前项目目标顺序

当前项目优先目标为：

1. 建立可维护、可审计、可持续扩展的知识资产。
2. 优先服务续写与改写。
3. 角色扮演及互动式用例后置，在续写与改写能力基本稳定后再推进。

## 7. 总原则

- 基座优先稳定，不随小说内容漂移。
- 模具只定义结构，不直接承担具体内容。
- 本地 LLM 负责首轮实例化，但其产出默认视为待审材料。
- 审核通过前，不得将首轮生成结果视为真值。
- 长期资产、实验资产与消费产物必须严格分层，不得混用。
- 分区只用于变化内容，不用于无差别拆分全部资产。

## 8. 资产层级 Tag 规范

### 8.1 目的

为确保 AI 与工具链能够稳定识别文件所属资产层级，除脚本文件外，项目内文件都应包含标准化的资产层级 Tag。

资产层级 Tag 仅用于声明文件所属层级，不用于声明内容正确性、审核状态、真值等级、权限细则或其他附加语义。
各层级的详细定义、职责与约束，以本规范正文为准，不由 Tag 本身承载。

### 8.2 适用范围

本规则适用于除脚本文件外的项目文件。
脚本文件本身不强制写入资产层级 Tag，但脚本生成的非脚本文件应遵循本规则。

### 8.3 合法层级值

资产层级 Tag 的取值只能是以下四个之一：

- `base`
- `instantiation_mold`
- `novel_instance`
- `experimental`

除上述四个值外，均视为非法值。

### 8.4 JSON 类文件字段名规范

对于 JSON 类文件，资产层级 Tag 的唯一合法字段名为 `_asset_layer`。

标准写法如下：

```json
{
  "_asset_layer": "base"
}
```
